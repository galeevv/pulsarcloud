# Security

OTPs, email magic-link tokens, and session/completion/start tokens are never stored in plaintext or logged. Verification values use keyed HMAC; credentials needed by the delivery outbox and payout details use AES-256-GCM with a separate 32-byte key. Email magic links and Telegram completion links are additionally bound to signed, per-challenge browser-state cookies, so a missing or mismatched browser cannot consume the token. Identity uniqueness is enforced in SQLite, Telegram authorization uses numeric ID only, and account merge is forbidden.

Cookies are HttpOnly and SameSite=Lax; production uses Secure host-only `__Host-` names with `Path=/` and no `Domain`, preventing a sibling subdomain from tossing a parent-domain session cookie. Cookie-authenticated Route Handler mutations require an exact `Origin` match to `APP_URL`, `Sec-Fetch-Site: same-origin` when supplied, and JSON content type where a body is expected. Server Actions retain Next.js' own origin protections and re-authorize every mutation. Telegram and Platega webhooks use provider secrets instead of browser CSRF. Provider bodies are bounded. Technical integration details stay in `IntegrationLog`/job errors and friendly text is shown to users.

Production checklist: rotate all example values; verify Resend sender; use random webhook secrets; restrict `/etc/pulsar/pulsar.env` to root/pulsar; expose only 22/80/443; bind app and Remnawave to loopback; keep test mode off; patch Node/npm dependencies; back up before migrations; monitor audit/job logs. The emergency production test override accepts only a local SQLite URL whose canonical filename is explicitly `test.db`, `*.test.db`, `test-*.db`, or `*-test.db`; symlinks, URL parameters, remote file URLs, and incidental `test` substrings are rejected, and test/real identity parity is still enforced. `npm audit` findings must be reviewed before each release rather than force-upgraded blindly.
