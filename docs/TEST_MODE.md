# Test mode

Test mode is adapter-driven: local and production run the same Next.js routes, domain services, Prisma schema, session logic, worker, and outbox code. Configuration selects only the external adapters and test-only surfaces; there is no second mock application to drift from production. Test mode exposes dev OTP, `/test/checkout/[id]`, `/api/test/payments/[id]`, a persistent dashboard banner, and `/admin/test`. The admin test page covers marked users and OTP, payment confirmation plus duplicate event, Telegram login, referral registration plus first payment, payout creation, provisioning success/failure, accelerated expiry, and confirmed deletion restricted to non-admin test users. These actions are audited.

Enable locally with `npm run setup:local`, then start web and worker together with `npm run dev:local` (or its `npm run dev:all` alias). The setup script enforces `PULSAR_TEST_MODE=true`, `PAYMENT_PROVIDER=test`, and `REMNAWAVE_PROVIDER=mock`, and refuses to modify an existing production or ambiguous env file. Configuration rejects any other provider combination in test mode and rejects test/mock adapters outside test mode. Test checkout, payout, and referral paths also enforce `User.isTest` parity, so an emergency production override cannot credit or charge an existing real user through a test adapter. Disable by setting `PULSAR_TEST_MODE=false`; test pages then call `notFound` and OTP is never returned.

Production startup fails when test mode is true unless `PULSAR_ALLOW_TEST_MODE_IN_PRODUCTION=true`; even then, startup requires a local SQLite `DATABASE_URL` whose canonical filename is explicitly `test.db`, `*.test.db`, `test-*.db`, or `*-test.db`. Symlinks, URL parameters, remote file URLs, and incidental substrings such as `latest.db` are rejected. Auth/billing/referral/payout paths also enforce test-user parity. The override remains an emergency foot-gun and must never be enabled on the public service.
