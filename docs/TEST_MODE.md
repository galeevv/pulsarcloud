# Test mode

Test mode is adapter-driven: local and production run the same Next.js routes, domain services, Prisma schema, session logic, worker, and outbox code. Configuration selects only the external adapters and test-only surfaces; there is no second mock application to drift from production. Test mode exposes `/test/checkout/[id]`, `/api/test/payments/[id]`, a persistent dashboard banner, and `/admin/test`; non-production test mode additionally returns dev OTP and enables the local Telegram simulator. The admin test page covers marked users and OTP, payment confirmation plus duplicate event, Telegram login, referral registration plus first payment, payout creation, provisioning success/failure, accelerated expiry, and confirmed deletion restricted to non-admin test users. These actions are audited.

Enable locally with `npm run setup:local`, then start web and worker together with `npm run dev:local` (or its `npm run dev:all` alias). The default setup enforces `PULSAR_TEST_MODE=true`, `PAYMENT_PROVIDER=test`, `REMNAWAVE_PROVIDER=mock`, and `PULSAR_ALLOW_LIVE_REMNAWAVE_IN_TEST_MODE=false`; it refuses to modify an existing production or ambiguous env file. Test checkout, payout, and referral paths also enforce `User.isTest` parity, so an emergency production override cannot credit or charge an existing real user through a test adapter. Disable by setting `PULSAR_TEST_MODE=false`; test pages then call `notFound` and OTP is never returned.

Live local provisioning is an explicit exception for integration testing. Create a separate short-lived token in the protected Remnawave operator UI, store it only as `REMNAWAVE_API_TOKEN` in the ignored root `.env`, then run `npm run setup:local:remnawave`. The current Panel API tokens have full API access rather than granular user-only scopes, so revoke the token immediately after testing. This keeps `PAYMENT_PROVIDER=test`, switches only provisioning to `REMNAWAVE_PROVIDER=http`, enables `PULSAR_ALLOW_LIVE_REMNAWAVE_IN_TEST_MODE=true`, and uses an isolated namespace (`pulsar_local_test` by default). The namespace participates in the deterministic remote username hash, so a local user ID cannot address the production user's Remnawave record. The setup validates the HTTPS endpoint, token presence, distinct squad UUIDs, and namespace before writing or backing up `.env`. Start both web and worker with `npm run dev:local`; provisioning is performed by the worker. Running `npm run setup:local` restores the mock provider but cannot revoke the remote Panel token; revoke it in the Panel UI and clear `REMNAWAVE_API_TOKEN` when the test is complete.

Production startup fails when test mode is true unless `PULSAR_ALLOW_TEST_MODE_IN_PRODUCTION=true`; even then, startup requires a local SQLite `DATABASE_URL` whose canonical filename is explicitly `test.db`, `*.test.db`, `test-*.db`, or `*-test.db`. Symlinks, URL parameters, remote file URLs, and incidental substrings such as `latest.db` are rejected. Auth/billing/referral/payout paths also enforce test-user parity. The override remains an emergency foot-gun and must never be enabled on the public service.

`PULSAR_ALLOW_LIVE_REMNAWAVE_IN_TEST_MODE=true` is rejected when test mode is off or when the namespace is the production default `pulsar`. A public production-runtime test deployment is allowed only when the separate production override is also enabled and the database has an explicitly test-only canonical filename; use an isolated namespace such as `pulsar_vps_test`. This triple opt-in is for controlled acceptance testing only and must be removed before real billing is enabled.

For the controlled VPS acceptance environment, run `deploy/pulsar/enable-vps-test-mode.sh` as root before migrations. It preserves the protected production environment as a timestamped backup, switches to `/var/lib/pulsar/pulsar-vps-test.db`, keeps payments on the test adapter, and uses the live Panel only through the isolated `pulsar_vps_test` namespace. In this production-runtime test mode, authentication still uses real Resend and Telegram Bot API gateways; dev OTP and the local Telegram simulator remain limited to non-production development. After cutover, run `deploy/pulsar/audit-vps-test-mode.sh`; unlike the strict production audit, it expects the explicit test flags and verifies database isolation, loopback listeners, public health, blackhole entitlement fixtures, and the absence of a traffic Node.
